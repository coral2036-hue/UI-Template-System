작업 배경 & 목표

🎯 왜 이 작업을 했나
HTML 프로토타입(통합_시연용_v9.html)으로 기능과 시나리오를 먼저 검증한 후, 실제 개발에 넘기기 위한 정식 Figma 디자인이 필요했습니다.

단순한 목업이 아니라 "초급 React 개발자가 Figma만 보고 개발·디버깅 가능한 수준"이 목표였습니다.
📋 작업 범위
• 해상도: 1440×900 (데스크탑 기준)
• 채팅/응답 화면: A4 세로 비율, 높이 자유 확장
• 시나리오 6개 전체 (일반/분석/예측/이상거래/상담/보고서)
• 모든 이벤트 상태 포함 (등록/수정/삭제/전송/로딩)
• 각 화면 옆 개발자 가이드(DESC) 프레임 별도 작성
• 최종 React 개발 프롬프트 템플릿 2개 파일 납품

Figma 작업 방법 — Plugin API 활용

일반적인 Figma 수동 작업이 아닌, Plugin API를 JavaScript로 직접 호출하는 방식으로 대량 프레임을 자동 생성했습니다.

⚙️ 작업 방식: use_figma (JavaScript → Figma Plugin API)
Figma 앱 내부에서 실행되는 Plugin API에 JavaScript 코드를 전달하는 방식입니다.
프레임 생성, 텍스트 삽입, 색상/폰트 설정, 위치 지정까지 모든 작업을 코드로 처리했습니다.

작업 단위 예시:
  • 프레임 1개 생성 → 내부에 Sidebar / Header / ChatArea 프레임 분리
  • ChatArea 안에 User-1, AI-1, User-2, AI-2 ... 순서로 Turn 프레임 삽입
  • 각 Turn 안에 report-header, number-stat × 4, data-table, bar-chart 등 블록 삽입
  • 텍스트 노드 폰트·크기·색상·정렬 개별 설정
✅ Plugin API 방식의 장점
• 수백 개 노드를 한 번에 생성 가능
• HTML CSS 값을 그대로 Figma 속성으로 변환
• 위치(x, y)를 수치로 정밀하게 제어
• 반복 수정 시 코드 재실행으로 일괄 적용
• 데이터 기반 실제 수치를 텍스트에 자동 삽입
⚠️ Plugin API 방식의 한계
• 한글 폰트 로드 실패 시 렌더링 오류
• Auto Layout 미지원 → 수동 y값 계산 필요
• 콘텐츠 높이를 미리 알 수 없어 겹침 발생
• 스크린샷으로 확인 후 재수정하는 반복 필요
• Figma 커넥터 인증 만료 시 작업 중단
⚠️ 토큰 소모 — 실제 과금 구조 이해 (중요)
❌ 잘못된 이해 (콘텐츠 기준)
"이번 대화에서 생성한 텍스트·코드의 총량"
→ 약 300,000 토큰

이건 실제 청구와 무관한 수치입니다.
✅ 실제 과금 구조 (누적 context)
매 턴마다 대화 전체 기록이 input으로 전송됨
→ 턴이 쌓일수록 input 토큰이 눈덩이처럼 증가

실제 청구 기준: 약 1,000~1,500만 토큰
계산 근거: 이 작업은 약 150~200턴 진행, 컨텍스트 압축(compaction) 1회 발생.
압축 전 평균 context ~80,000 토큰 × 100턴 = 8,000,000 / 압축 후 ~60,000 × 50턴 = 3,000,000 / output ~500,000
→ 총 약 11,500,000 토큰 (Claude API input+output 합산 기준)

주요 수정 과정 — 발견 → 원인 → 해결

스크린샷으로 확인 → 문제 발견 → 코드 수정 → 재확인 사이클을 반복했습니다.

폰트 문제
한글이 전혀 보이지 않음 — Inter 폰트 렌더링 실패
현상: 첫 스크린샷 확인 시 한글 텍스트가 모두 빈 칸으로 표시됨
원인: Figma Plugin API 기본 폰트인 Inter가 한글 문자를 지원하지 않음. 코드에서 fontName을 명시하지 않으면 Inter로 fallback됨
해결: 모든 한글 텍스트 노드에 { family: "Noto Sans KR", style: "Regular" } 명시적으로 지정. 기존 생성된 715개 노드를 일괄 순회하며 폰트 교체
교훈: Plugin API에서 한글 폰트는 반드시 fontName을 명시해야 함
레이아웃 문제

6

겹침이 반복 발생한 구조적 원인

Plugin API 방식으로 Figma를 그릴 때 겹침이 반복된 이유는 단순 실수가 아닌 구조적 한계에서 비롯됐습니다.

🔴 원인 1 — Auto Layout 미지원: 모든 y값을 사람이 계산
일반 Figma 작업에서는 Auto Layout을 켜면 자식 요소가 쌓일 때 자동으로 간격을 유지합니다.
Plugin API에서는 Auto Layout을 신뢰하기 어려워 모든 요소의 y 좌표를 코드에서 직접 지정해야 했습니다.

예: 블록A.y = 100, 블록B.y = 100 + 블록A.height + 16 — 블록A 높이가 바뀌면 블록B 이후 모든 값을 재계산해야 합니다.
→ 하나의 블록 높이가 예상과 다르면 그 아래 모든 요소가 연쇄적으로 잘못된 위치에 생성됩니다.
🔴 원인 2 — 텍스트 높이는 삽입 전까지 알 수 없음
텍스트 블록의 실제 높이는 폰트 크기, 줄바꿈, 텍스트 길이에 따라 달라집니다.
코드 실행 전 정확한 height를 예측할 방법이 없어, 추정값으로 다음 y를 계산했습니다.

예: DESC 텍스트 블록을 400px로 예상했지만 실제 렌더링은 620px → 하단 요소 220px 겹침 발생.
→ 해결하려면 생성 후 node.height를 읽어 다음 요소 y를 재계산하는 2-pass 방식이 필요한데, 이 자체가 추가 코드·토큰·시간을 소모합니다.
🔴 원인 3 — 멀티턴 채팅의 연쇄 의존성
S1 화면처럼 5턴 대화가 순서대로 쌓이는 구조에서, 앞 턴의 AI 응답 높이가 달라지면 뒤따르는 모든 턴의 y가 틀려집니다.

실제로 AI-2 응답이 예상(~600px)보다 길게(704px) 생성되어, User-3이 AI-2 내부로 들어가는 문제가 발생했습니다.
10개 턴 전체를 다시 순회하며 y값을 재계산하고 재배치하는 작업이 필요했습니다.
→ 연쇄 의존 구조일수록 한 곳의 오차가 전체에 전파됩니다.
🟡 원인 4 — DESC 프레임이 화면 높이에 종속
각 화면 옆에 배치된 DESC 프레임은 해당 화면의 높이를 기준으로 x 위치를 정합니다.
화면 높이가 수정될 때 DESC 위치를 함께 업데이트하지 않으면 DESC가 다른 행의 프레임과 겹치는 현상이 발생했습니다.

특히 S1(4,071px), S2(2,150px)처럼 높이가 큰 화면에서 DESC가 3행 프레임(S4~S6)의 y 영역까지 침범하는 경우가 있었습니다.
🟡 원인 5 — 대량 생성 후 몰아서 확인
여러 프레임을 한꺼번에 생성한 뒤 스크린샷으로 확인하는 방식이다 보니, 문제를 발견했을 때는 이미 여러 겹의 오류가 누적된 상태였습니다.

프레임 A의 겹침을 수정하다 보면 프레임 B의 위치도 영향을 받아 새 겹침이 생기는 악순환이 반복되었습니다.

가장 어려웠던 부분 TOP 3

🏆 1위 — 콘텐츠 높이의 불확실성 (겹침 문제의 근본 원인)
Figma Plugin API는 Auto Layout을 지원하지 않습니다. 모든 요소의 y값을 코드에서 직접 계산해야 합니다.

문제는 텍스트 블록의 실제 높이를 코드 실행 전에 알 수 없다는 것입니다. 텍스트 길이, 줄바꿈, 폰트 크기에 따라 높이가 달라지는데, 이를 사전에 예측하고 다음 요소의 y값을 정확히 계산하는 것이 구조적으로 어렵습니다.

해결 방법: 생성 후 코드로 실제 height를 읽고, 그 값을 다음 요소 y 계산에 반영하는 2단계 방식. 하지만 한 번에 완성되지 않고 스크린샷 확인 → 수정의 반복이 불가피했습니다.
🥈 2위 — 한글 폰트 관리
Figma Plugin API에서 폰트를 사용하려면 figma.loadFontAsync()를 명시적으로 호출해야 합니다. 한글의 경우 Noto Sans KR을 로드해야 하는데, 이를 빠뜨리면 텍스트 생성 시 에러가 발생하거나 빈 텍스트가 렌더링됩니다.

초기에 Inter를 기본 폰트로 사용했다가 한글이 전혀 표시되지 않는 문제를 뒤늦게 발견했고, 이미 생성된 715개 노드를 전부 순회하며 폰트를 교체하는 작업이 필요했습니다.

추가 어려움: 숫자/영문은 Inter, 한글은 Noto Sans KR을 써야 하는 경우 동일한 텍스트 노드에 mixed font를 적용하기 어려워 노드를 분리해야 했습니다.
🥉 3위 — 멀티턴 채팅 화면의 총 높이 관리 (S1 = 4,071px)
S1 일반질의 화면은 5턴 멀티턴 대화로 총 높이가 4,071px입니다. 각 AI 응답에는 report-header, number-stat ×4, data-table, bar-chart, callout, steps, source-box 등 수십 개의 블록이 들어갑니다.

한 턴의 높이가 달라지면 그 이후 모든 턴의 y값이 연쇄적으로 바뀌어야 합니다. 이를 수동으로 계산하다가 User-3가 AI-2 안에 들어가는 문제가 발생했고, 결국 10개 턴 전체를 코드로 재배치하는 작업이 필요했습니다.

💡 왜 38배나 차이나나? — API는 매 턴마다 대화 전체를 input으로 보냄. 150턴 대화에서 평균 context가 80,000이면 input만 1,200만 토큰. 컨텍스트 압축(compaction)이 1회 발생했다는 것 자체가 200,000 토큰 한도에 도달했다는 증거.

🤝 역할 분담 — AI 자동화 vs 사람이 직접 한 것
🤖 Claude AI (Plugin API 자동화)
• 40개 프레임 구조 생성
• 텍스트 노드 1,500개+ 삽입

• 한글 폰트 715개 노드 일괄 교체
• DESC 11개 내용 작성
• 차트 바 텍스트 중앙 정렬 수식 계산
• ShareRow / NS카드 규칙 통일 코드 실행
• 프롬프트 템플릿 파일 2개 작성
👤 담당자 직접 수정
• 간격·여백 미세 조정 (눈으로 보고 직접 조율)
• 일부 텍스트 오탈자 및 문구 수정
• 결과물 검토 후 추가 요구사항 피드백
• ShareRow 버튼 배치 방향성 지시
• number-stat 카드 참고 이미지 제공
• 이상거래 테이블 정렬 방향 직접 지시
• 전체 완성도 기준 최종 판단 및 승인
💡 Plugin API 자동화는 구조 생성과 대량 반복 작업에 강했고, 사람이 직접 한 부분은 디자인 감각이 필요한 미세 조정과 방향성 결정이었음. 두 역할이 번갈아 가며 품질을 높임.
💪 잘 된 것
• Plugin API를 통한 대량 프레임 자동 생성으로 수작업 대비 시간 대폭 단축
• HTML CSS 변수 → Figma 토큰 1:1 매핑으로 일관성 확보
• 실제 데이터 기반으로 디자인해 현실적인 레이아웃 검증 가능
• 스크린샷 기반 반복 검증으로 주요 겹침 구조 파악
• 담당자 직접 세부 조정으로 최종 겹침 0건 완성
• DESC 프레임으로 디자이너-개발자 간 커뮤니케이션 비용 감소
• 담당자의 구체적인 피드백(이미지 첨부, 정렬 지시 등)으로 빠른 방향 수정 가능
🔄 다음에 다르게 할 것
• 처음부터 텍스트 높이를 2패스로 계산하는 함수 먼저 작성
• 폰트 로드 유틸 함수를 공통으로 만들어 모든 텍스트 생성에 적용
• 공통 컴포넌트(NS카드, ShareRow 등) 먼저 검증 후 복제
• 각 화면 생성 직후 바로 스크린샷 확인 (나중에 몰아서 X)
• 미세 조정이 필요한 구간은 처음부터 담당자와 실시간 확인하며 진행
• 프레임 배치 전 캔버스 좌표 계획을 표로 먼저 정리
