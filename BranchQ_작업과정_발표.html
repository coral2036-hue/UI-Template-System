01 · Background
작업 배경 & 목표
왜 이 작업을 했고, 무엇을 목표로 했나
🎯 작업 배경
HTML 프로토타입(통합_시연용_v9.html)으로 기능과 시나리오를 검증한 후, 실제 개발에 넘기기 위한 정식 Figma 디자인이 필요했습니다.
💡 단순 목업이 아닌 "초급 React 개발자가 Figma만 보고 개발·디버깅 가능한 수준"이 목표였습니다.
📋 작업 범위
• 해상도: 1440×900 데스크탑
• 채팅 화면: 높이 자유 확장 (A4 세로 비율)
• 시나리오 6개 전체 (일반/분석/예측/이상거래/상담/보고서)
• 모든 이벤트 상태 (등록/수정/삭제/전송/로딩)
• 각 화면 옆 개발자 가이드(DESC) 별도 작성
• React 개발 프롬프트 템플릿 2개 파일 납품

02 · Method
Figma 작업 방법 — Plugin API
수동 작업 없이 JavaScript로 Figma를 직접 제어
// Plugin API로 프레임 생성 예시
const frame = figma.createFrame();
frame.name = "S1-일반질의";
frame.resize(1440, 4071);

// 텍스트 노드 삽입
const t = figma.createText();
await figma.loadFontAsync(
  { family: "Noto Sans KR", style: "Bold" }
);
t.characters = "2억 3,840만원";
t.fontSize = 28;
✅ 장점
• 수백 개 노드를 한 번에 생성
• HTML CSS값을 Figma 속성으로 직접 변환
• 위치(x, y)를 수치로 정밀 제어
• 실제 데이터(수치) 기반 디자인 구현
⚠️ 한계
• Auto Layout 미지원 → y값 수동 계산 필수
• 텍스트 높이를 생성 전에 알 수 없음
• 한글 폰트 명시 필수 (안 하면 빈 텍스트)
• 확인은 스크린샷으로만 가능

04 · Iteration
주요 수정 과정
스크린샷 확인 → 문제 발견 → 코드 수정 → 재확인 사이클 반복
폰트 한글 텍스트 전체 빈칸 — Inter 폰트 한글 미지원
생성된 715개 노드를 전부 순회하며 Noto Sans KR로 교체. 처음부터 fontName 명시했으면 불필요한 작업.
겹침 프레임 간 대규모 겹침 — 6행 구조에서 반복 충돌
채팅 화면 높이가 콘텐츠에 따라 달라지면서 하단 프레임과 충돌. 6행→5행 재구성 후 전체 y값 재계산.
겹침 S1 멀티턴 — User-3이 AI-2 응답 안으로 들어감
AI-2 실제 높이(704px)가 예상보다 커서 이후 턴 전체가 잘못된 위치에 생성됨. 10턴 전체 y값 재배치.
정렬 S6 결재란 ↔ 보고서 제목 겹침 (3회 반복)
제목 width 미지정으로 전체 너비 차지. 제목 width=350px 제한으로 최종 해결.
정렬 S2 차트 "기타" 바 텍스트가 바 밖으로 이탈
바 center x = barX + barWidth/2 공식으로 정확한 중앙 정렬 적용. x=224 → x=136.

05 · Root Cause
겹침이 반복된 구조적 원인
단순 실수가 아닌 Plugin API 방식의 구조적 한계
🔴 Auto Layout 미지원
모든 y값을 코드에서 직접 계산해야 함.
블록A 높이가 달라지면 그 아래 모든 요소 y가 연쇄적으로 틀어짐.
🔴 텍스트 높이는 삽입 전 알 수 없음
폰트·줄바꿈·텍스트 길이에 따라 높이가 달라짐.
추정값으로 계산하면 실제와 달라져 겹침 발생.
🟡 멀티턴 연쇄 의존성
앞 턴의 높이가 달라지면 뒤따르는 모든 턴의 y가 틀어짐.
S1 5턴 → 오차 하나가 전체에 전파.
🟡 대량 생성 후 몰아서 확인
여러 프레임을 한꺼번에 만든 뒤 확인하면
발견 시점에 이미 오류가 여러 겹 누적된 상태.
⚠️ 최종 겹침 0건은 담당자가 직접 세부 조정해 달성한 결과입니다. Plugin API만으로는 겹침을 완전히 제거하기 어려운 구조적 한계가 있었습니다.

06 · Solution
겹침 이슈 개선 방안
다음에 동일 작업 시 적용할 전략
원인개선 방법예상 효과
Auto Layout 미지원 stackBlocks(parent, blocks, gap) 공통 함수 사전 작성 — 생성 후 height 읽어 자동 재배치 겹침 80% 감소
텍스트 높이 불확실 텍스트 삽입 직후 node.height 읽기 의무화 (2-pass 방식) 근본 해결
멀티턴 연쇄 의존 1턴 생성 → height 읽기 → 다음 턴 y 결정. 한 번에 5턴 생성 금지 연쇄 오류 차단
대량 생성 후 몰아 확인 프레임 1개 생성 직후 스크린샷 확인 → 다음 진행 누적 오류 방지
전체 공통 각 화면 완성 후 겹침 감지 코드 자동 실행 (bounding box 충돌 검사) 조기 발견
💡 핵심은 "생성과 검증을 분리하지 않는 것" — 생성 → 즉시 height 읽기 → 즉시 겹침 확인 → 다음 생성 루틴 준수

07 · Hardest Parts
가장 어려웠던 부분 TOP 3
🏆 1위
콘텐츠 높이의 불확실성
Auto Layout 없이 y값을 수동 계산해야 하는데, 텍스트 블록의 실제 높이는 생성 전에 알 수 없음.

예상(400px)과 실제(620px)가 달라지면 그 아래 모든 요소가 연쇄적으로 틀어짐.
🥈 2위
한글 폰트 관리
Plugin API에서 fontName을 명시하지 않으면 Inter로 fallback 되어 한글이 전혀 표시되지 않음.

이미 생성된 715개 노드를 전부 재처리해야 했음.
🥉 3위
S1 멀티턴 높이 관리
5턴 × AI응답 각 수백~700px = 총 4,071px.
한 턴의 높이가 바뀌면 이후 전체 턴 y값이 연쇄적으로 재계산 필요.

10개 턴 전체를 코드로 재배치.

08 · Limitations
기술적으로 안 된 부분 & 우회 방법
항목안 된 이유우회 방법
불가Auto Layout Plugin API에서 Auto Layout 속성 생성 시 의도대로 동작하지 않음 모든 y값을 코드에서 수동 계산
불가실시간 높이 예측 Figma는 노드를 실제 추가해야 height 값이 확정됨 생성 후 height 읽기 → 2-pass 방식
제한Component 재사용 Variant/Prop 관리가 복잡해 일관성 유지 어려움 각 화면에 독립 프레임으로 복제
제한Mixed 폰트 텍스트 일부에만 다른 폰트 지정하는 API가 복잡 노드를 숫자용/한글용으로 분리 배치
제한이미지 삽입 외부 이미지 URL 삽입이 제한됨 이모지로 아이콘 대체, 차트는 도형으로 직접 드로잉

• 한글 폰트 재처리 단독으로 ~8,000 토큰 소모 (사전 지정했으면 0)

10 · Collaboration
역할 분담 — AI 자동화 vs 담당자 직접
🤖 Claude AI (Plugin API 자동화)

• 텍스트 노드 1,500개+ 삽입
• 한글 폰트 715개 노드 일괄 교체

• 전체 완성도 기준 최종 판단 및 승인
