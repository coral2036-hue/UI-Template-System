테스트 진행 조건

보고서 목적

Claude Code를 활용한 UI 디자인 기획 작업에서, Figma와 Pencil.dev 두 도구의 효율성을 비교합니다. 동일한 원본 화면(통합_시연용_v9.html, Branch Q 페이지)을 동일한 프롬프트 구조로 두 도구에 각각 작업한 결과물을 분석하여, 향후 하나의 도구로 통일하기 위한 판단 근거를 제공합니다.

작업 프롬프트 원문

● Figma용 프롬프트

당신은 이제 UI/UX 기획자로 개발자에게 전달하기 위한 디자인 작업을 할 것입니다. https://www.figma.com/design/moHy3BaQLmzoGcXTgbJeI1/claude-code?node-id=0-1 이 피그마 링크에 브랜치Q 라는 페이지에 디자인을 그려야 합니다. 첨부한 html 파일을 가지고만 디자인 작업을 하여야 하며, 좌측, 상단의 프레임은 이미지를 참고하세요. 당신이 모든것을 그려야 하기 때문에 최대한 모든 구성요소를 디자인 하여야 하며, 각 화면별 디스크립션을 최대한 자세히 적어 주어야 합니다 (화면 우측 영역 별도로 하여) 모든 프레임은 겹침이 없어야 합니다. 화면 개발은 전체 리엑트로 진행되니 참고하세요. 임의로 없는 것을 만들지 말고 초급 개발자가 해당 페이지의 내용만으로 개발 및 디버깅 가능한 수준으로 작업 해 주세요

왜 화면 수가 이렇게 다를까?

● Figma의 접근 방식

"시나리오마다 따로 그린다"

  • 일반질의, 분석질의 등 6개 시나리오를 각각 실제 데이터로 화면 제작
  • 개발자가 "이런 데이터일 때 이렇게 보인다"를 눈으로 바로 확인
  • 상태 변형(열림/닫힘, 빈 상태/입력 상태)도 별도 화면으로 제공
  • 화면이 많아 관리 부담 증가
  • 수정 시 여러 화면을 일괄 업데이트해야 함

● Pencil의 접근 방식

"하나의 틀로 여러 상황을 표현한다"

  • ChatMode 1장으로 6개 시나리오를 구조적으로 커버
  • BlockRenderer 부품을 채팅/편집기/모바일에서 재사용
  • 관리 간편, 수정할 곳 적음, AI 비용 절약
  • 시나리오별 실제 모습을 시각적으로 보여주지 못함
  • 개발자가 "이 경우엔 어떻게 보이지?" 상상해야 함

왜 Pencil이 1회당 비용은 더 비싼데 총비용은 저렴할까?
Pencil MCP는 매 대화마다 시스템 프롬프트(~15,000토큰)와 스키마(~15,000토큰)를 보내야 해서 1회당 비용이 높습니다. 하지만 Flexbox 자동 정렬 덕분에 겹침 수정이 거의 필요 없어 총 대화 횟수가 66% 줄어들었고, 결과적으로 총비용은 33% 절약되었습니다. 즉, "1회는 비싸지만 횟수가 적어서 총합이 저렴한" 구조입니다.

발생한 문제 비교

Figma 발생 문제 (8건)

  1. 한글이 안 보임 — 기본 폰트(Inter)가 한글 미지원. 715개 텍스트 노드를 Noto Sans KR로 일일이 교체
  2. 화면 대규모 겹침 — 6행 배치 구조에서 DESC 프레임과 시나리오 프레임이 충돌. 5행으로 전면 재구성
  3. 채팅 말풍선 겹침 — S1 화면의 5턴 대화에서 AI 응답이 예상보다 커서 다음 메시지와 겹침. 10개 턴 위치 전부 재계산
  4. 숫자 카드 텍스트 겹침 — 값 텍스트(28px)와 라벨(13px)이 같은 위치. 13개 카드 모두 수정
  5. 보고서 제목 vs 결재란 겹침 — S6 화면에서 가로 방향 충돌. 3번 반복 후 해결 (제목 너비 제한 350px)
  6. 차트 글자 넘침 — "기타(8%)" 텍스트가 바 영역 밖으로 넘침. 중앙 정렬 공식 적용
  7. 공유 버튼 크기 불일치 — PDF/Excel/Share/Report 버튼 높이/너비/테두리가 제각각. 전체 통일(h=36, min-w=90, radius=20)
  8. 이상거래 표 정렬 — 금액이 왼쪽 정렬, 위험점수 글자 너무 작음. 오른쪽 정렬 + 24px 볼드로 수정

Pencil 발생 문제 (1건)

  1. 전체 화면 배치 겹침 — 캔버스 레벨에서 프레임들의 Y좌표가 겹침. 9개 프레임의 Y축 재배치로 해결
Pencil이 문제가 적었던 이유:
1) Flexbox(자동 정렬) 기반이라 내부 요소 겹침이 구조적으로 발생하지 않음
2) 추가 수정 요청을 거의 하지 않았으므로 수정 과정에서 발생하는 새 문제도 없었음
3) 한글 폰트 문제가 없어 초기 설정 이슈 없음
공통 이슈: 두 도구 모두 캔버스 레벨의 프레임 겹침이 발생했으며, 최종적으로 사용자가 직접 확인하고 수정하는 과정이 필요했습니다. AI만으로는 완벽한 배치가 어렵다는 공통적 한계입니다.

도구별 장단점 정리

● Figma

좋은 점

  • 업계 표준 — 디자이너/개발자 누구나 익숙함
  • 시나리오마다 실제 데이터로 개별 화면 제공
  • 링크 하나로 외부 공유, 브라우저에서 바로 확인
  • Dev Mode로 개발자가 수치/색상 직접 확인
  • 풍부한 플러그인/핸드오프 도구 생태계
  • 실시간 댓글로 팀 소통 가능
  • 버전 이력 자동 관리
  • 설명서(DESC)가 매우 상세 — 신입도 독립 개발 가능

아쉬운 점

  • AI 비용이 높음 (~1,150만 토큰)
  • 전체 작업의 ~40%가 수정에 소비
  • 한글 폰트 문제 (초기 설정 필요)
  • 절대 좌표 방식 → 겹침이 잦음
  • 대화 ~150-200회로 오래 걸림

● Pencil.dev

좋은 점

  • Flexbox 자동 정렬 → 겹침 거의 없음
  • 대화 ~50-60회로 빠르게 완성 (66% 감소)
  • 총 비용 33% 절약
  • 수정이 거의 불필요 (~5%)
  • 23개 색상 변수 체계적 관리
  • Claude Code와 MCP 직접 연동 — 중간 단계 적음
  • 블록 타입 16종으로 Figma보다 2종 더 많이 정의

아쉬운 점

  • 화면 17장으로 시나리오별 개별 화면 없음
  • 업계 비표준 — 새로 배워야 함
  • .pen 파일 전달 필요 → 외부 공유 어려움
  • 실시간 협업/댓글 기능 없음
  • 플러그인/연동 도구 생태계 제한적
  • 설명서에 라우트/API/에러상태 없음
  • 이번 테스트에서 수정을 많이 안 시켜서 수정 내구성 미검증

수정 / 신규 추가 시 어떤 게 더 유리한가?

이미 만든 화면을 수정할 때

관점FigmaPencil.dev
수정 방법 Plugin API로 특정 노드 지정 수정 batch_design의 Update 연산
수정 후 부작용 다른 요소와 겹칠 위험 있음 (절대 좌표) 자동 정렬로 부작용 적음 (Flexbox)
검증 방법 스크린샷 → 육안 확인 반복 snapshot_layout 자동 겹침 검증
판정: 수정 안정성은 Pencil이 우위. 자동 정렬 덕분에 한 곳을 고쳐도 다른 곳에 영향이 적습니다.

새 기능/화면을 추가할 때

관점FigmaPencil.dev
기존 화면 수 40장 → AI가 기억할 내용 많음 17장 → 상대적으로 여유
추가 비용 예상 높음 (컨텍스트 누적 ~100K/턴) 중간 (컨텍스트 여유 있음)
추가 품질 실제 데이터 기반 개별 화면 → 구체적 기존 틀 재사용 + DESC 추가 → 빠르지만 추상적
판정: 확장성(추가 작업의 가벼움)은 Pencil이 유리하고, 추가된 화면의 상세함은 Figma가 유리합니다.

계속 수정을 반복할 때

관점FigmaPencil.dev
수정 비율 ~40% (이번 테스트 기준) ~5% (수정 요청 적었음)
수정 시 비용 증가 눈덩이식 — 수정할수록 비용 급증 완만 — 자동 정렬로 재작업 최소
수정 내구성 검증 검증됨 (8건 이슈 모두 해결) 미검증 (수정 요청 적어 데이터 부족)
유의: Pencil의 수정 비율 5%는 수정 요청을 적게 했기 때문입니다. 동일 수준의 수정을 요청하면 이 비율이 올라갈 수 있으며, 이를 확인하려면 추가 검증 테스트가 필요합니다.

Claude Code만으로 계속 작업해도 되는가?

작업 흐름 비교

● Figma 작업 흐름

Claude가 JS 코드 생성 → Figma Plugin API 실행 → 스크린샷 확인 → 수정 반복
  • 중간에 "코드 생성 → 실행" 변환 단계가 있어 왕복이 많음
  • 스크린샷 기반 검증 → 이미지 토큰 소비 (~1,500,000)
  • Figma 생태계의 모든 기능 활용 가능

● Pencil 작업 흐름

Claude가 MCP 직접 호출 → batch_design 실행 → 자동 검증
  • MCP 직접 호출 → 중간 변환 없이 빠름
  • snapshot_layout으로 자동 겹침 검증
  • Pencil 특화 MCP이므로 생태계 제한적

Claude Code 단독 작업의 한계

영역Claude가 할 수 있는 것사람이 해야 하는 것
화면 구조 생성프레임, 텍스트, 색상, 아이콘 배치최종 미세 정렬 확인
겹침 해결자동 감지 + 좌표 재계산시각적으로 자연스러운지 최종 확인
디자인 품질규칙 기반 일관성 유지디자인 감각 기반 미적 판단
데이터 입력구조적 데이터 배치비즈니스 맥락에 맞는 데이터 선별
추천 방식: Claude Code로 80% 빠르게 생성하고, 사람이 20% 마무리하는 협업 방식이 가장 효과적입니다. 두 도구 모두 이 패턴이 적용되며, 차이는 "80%까지 가는 속도"(Pencil이 빠름)와 "80%의 품질"(Figma가 높음)입니다.

다른 사람에게 공유하기 편한가?

공유 항목FigmaPencil.dev
링크 공유 바로 가능 — URL 하나로 전달 불가 — .pen 파일 전달 필요
설치 없이 보기 가능 — 브라우저에서 바로 불가 — Pencil.dev 설치 필요
개발자 수치 확인 Dev Mode — 크기/색상/간격 즉시 확인 파일 열어야 함

결론 및 추천

1순위 추천

Figma + Claude Code

  • 외부 공유가 압도적으로 편함 — 링크 하나로 누구든 바로 확인. 고객사/협력사/타 부서와의 소통에 필수
  • 개발 설명서 품질이 높음 — 라우트, API, 에러상태까지 포함. 신입 개발자도 독립 구현 가능
  • 시나리오별 실제 데이터 화면 — "이 경우 이렇게 보인다"를 눈으로 확인, 개발자 해석 부담 최소
  • 업계 표준 — 디자이너, PM, 개발자 누구나 쓸 줄 알아 학습 비용 없음
  • 풍부한 생태계 — Dev Mode, 플러그인, Zeplin 등 연동 도구 다수
2순위 대안

Pencil.dev + Claude Code

  • 비용 33% 절약 + 대화 66% 감소 — 예산/시간 제약이 클 때 유효
  • 배치 안정성이 뛰어남 — Flexbox 자동 정렬로 겹침 문제 거의 없음
  • 수정 시 부작용 적음 — 자동 정렬 덕분에 한 곳 고쳐도 다른 곳 안 깨짐
  • 빠른 초안 작업에 적합 — 내부 검토용 빠른 프로토타입 제작
  • 확장성 유리 — 기존 17장이라 새 화면 추가 시 AI 부담 적음

하나만 쓸지, 나눠 쓸지?

시나리오추천 도구이유
외부 공유가 필요한 경우 Figma 링크 공유, Dev Mode, 댓글 기능 필수
개발자에게 완성된 기획서 전달 Figma 라우트/API/에러상태 포함 상세 DESC
빠른 내부 검토용 초안 Pencil 빠르고 저렴하게 구조 확인
비용이 최우선 제약 Pencil 33% 절약, 66% 시간 단축
잦은 수정이 예상되는 경우 Pencil 자동 정렬로 수정 부작용 최소 (단, 추가 검증 필요)
하나만 선택해야 한다면 Figma 비용은 높지만 공유/협업/품질에서 종합 우위

보완이 필요한 점

● Figma 개선 사항

  • 겹침 문제 줄이기 — 프롬프트에 Auto Layout 우선 사용 지시, 좌표 재계산 로직 강화
  • 한글 폰트 사전 설정 — 작업 시작 전 Noto Sans KR 로드를 프롬프트에 포함
  • 수정 비용 줄이기 — 겹침 자동 감지 + 자동 수정 스크립트 사전 준비

● Pencil 검증 필요 사항

  • 수정 내구성 테스트 — 동일 수준의 수정 요청을 반복하여 수정 비율 재측정
  • 시나리오별 개별 화면 생성 — 지시하면 Figma 수준의 상세 화면 가능한지 확인
  • DESC 품질 향상 — 라우트, API, 에러상태 포함 여부 확인
  • 외부 공유 방안 — 이미지 내보내기 + 별도 문서화로 보완 가능한지
최종 요약: 종합적으로 Figma를 1순위로 추천합니다. 비용은 높지만 "만든 결과물을 다른 사람과 함께 볼 수 있고, 개발자가 바로 쓸 수 있는 품질"이 실무에서 더 중요하기 때문입니다. Pencil.dev는 2순위로, 빠른 초안이나 내부 검토 목적이라면 비용/시간 면에서 충분히 활용 가치가 있습니다. 다만, Pencil에 동일 수준의 수정 요청을 했을 때의 내구성은 추가 검증이 필요합니다.

결론에 미치는 영향

Pencil + 가이드 = 실질적 대안이 됨

  • DESC에 라우트/API/에러가 들어가면 신입 개발자도 독립 개발 가능
  • 시나리오별 화면을 만들면 Figma와 동등한 상세도
  • 비용 우위(33% 절약)는 여전히 유효
  • 배치 안정성은 여전히 Pencil이 우수
  • 외부 공유 문제는 해결 안 됨 (도구 자체의 한계)
  • 가이드 투입 = 사전 준비 비용 발생 (문서 작성 시간)

향후 검증 추천 사항

  • Pencil + 6개 가이드 MD + 동일 수정 횟수로 재테스트
  • Figma도 가이드 MD를 초기에 투입하여 수정 비율 변화 측정
  • 두 도구 모두 "가이드 있음/없음" 2x2 매트릭스로 비교
  • 외부 공유가 불필요한 프로젝트에서 Pencil + 가이드 조합 실전 테스트
요약: 가이드 문서는 "도구의 약점을 보완하는 보조 장치"입니다. Pencil에 투입하면 Figma와의 품질 격차가 크게 줄어들어 비용/시간 우위를 살린 실질적 대안이 됩니다. 다만 외부 공유가 필요한 프로젝트에서는 가이드를 투입해도 Figma를 대체하기 어렵습니다. 최종 추천은 변하지 않지만, Pencil의 2순위 활용 가치는 크게 올라갑니다.
